# Future Database Design (Planning)

This document plans the **database** the project will use in a later week
(Week 4). It is a **design**, not an implementation — there is **no SQL here**.
The goal is to decide *which tables will exist, what each holds, and how they
relate*, so that when we do build the database the structure is already thought
through.

It builds on the [`logistics_data_model.md`](logistics_data_model.md) mapping
and on the relational thinking from Week 0 (note 07: primary keys, foreign
keys, joins).

---

## Why a database at all?

Right now the data lives in CSV files we load with pandas. That is fine for
analysis, but a real platform needs a **database** so that:

- many parts of the system (API, dashboard, agents) can read/write the same
  data safely and at the same time;
- relationships are enforced (an order can't reference a customer who doesn't
  exist);
- queries stay fast as data grows.

We will use **PostgreSQL** (planned, Week 4). This document defines the tables.

---

## Quick vocabulary (from Week 0, note 07)

- **Table** — a grid of one kind of thing (e.g. all customers).
- **Primary key (PK)** — the column that uniquely identifies each row.
- **Foreign key (FK)** — a column pointing to another table's primary key.
- **Relationship** — how two tables connect (one-to-many, many-to-many).

## Table-by-table design

### 1. `customers` — delivery destinations
**Purpose:** every place an order must be delivered to.
**Important columns:**
- `customer_id` (PK) — per-order customer record.
- `customer_unique_id` — identifies the same real person across orders.
- `zip_code_prefix`, `city`, `state` — location.
- `latitude`, `longitude` — coordinates for distance/routing.
**Relationships:** one customer → many orders.
**Why it exists:** routing and delay analysis both need the destination.

### 2. `products` — inventory item catalog
**Purpose:** the catalog of items that can be ordered and stocked.
**Important columns:**
- `product_id` (PK).
- `category_name_english` — readable category.
- `weight_g`, `length_cm`, `height_cm`, `width_cm` — affect packing/freight.
**Relationships:** one product → many order items; one product → many inventory
records (one per warehouse).
**Why it exists:** weight/size feed freight and vehicle-capacity logic later.

### 3. `warehouses` — fulfillment origins (mapped from sellers)
**Purpose:** the locations shipments start from. Mapped from Olist *sellers*
(see logistics data model — this is our biggest modeling assumption).
**Important columns:**
- `warehouse_id` (PK) — from `seller_id`.
- `zip_code_prefix`, `city`, `state`.
- `latitude`, `longitude` — the route origin.
- `capacity` *(simulated later)* — how much it can handle per day.
**Relationships:** one warehouse → many order items; one warehouse → many
inventory rows.
**Why it exists:** every route and every stock level is anchored to an origin.

### 4. `orders` — shipment requests
**Purpose:** one customer's purchase = a request to deliver goods.
**Important columns:**
- `order_id` (PK).
- `customer_id` (FK → customers).
- `status` — delivered, shipped, canceled, etc.
- `purchase_timestamp`, `approved_at`, `delivered_carrier_date`,
  `delivered_customer_date`, `estimated_delivery_date`.
**Relationships:** one order → many order items; one order → one customer.
**Why it exists:** the central table; delays are measured here.

### 5. `order_items` — items within an order (the bridge table)
**Purpose:** links orders to the products and warehouses that fulfill them.
This is the **many-to-many bridge** between orders and products.
**Important columns:**
- (`order_id`, `order_item_id`) (composite PK).
- `order_id` (FK → orders), `product_id` (FK → products),
  `warehouse_id` (FK → warehouses).
- `price`, `freight_value`, `shipping_limit_date`.
**Relationships:** belongs to one order, one product, one warehouse.
**Why it exists:** it is the unit of physical movement (origin → destination).

### 6. `inventory` — stock levels per warehouse *(simulated, Week 2)*
**Purpose:** how many units of each product a warehouse currently holds.
**Important columns:**
- `inventory_id` (PK).
- `warehouse_id` (FK), `product_id` (FK).
- `stock_quantity`, `reorder_level`, `last_updated`.
**Relationships:** ties a product to a warehouse with a quantity.
**Why it exists:** out-of-stock events drive fulfillment and replanning
decisions. **Not in Olist — simulated.**

### 7. `routes` — planned delivery paths *(computed, Week 3)*
**Purpose:** the path/plan to move items from an origin to a destination.
**Important columns:**
- `route_id` (PK).
- `origin_warehouse_id` (FK), `destination_customer_id` (FK).
- `vehicle_id` (FK), `distance_km`, `estimated_time`, `sequence` (stop order).
**Relationships:** connects a warehouse, a customer, and a vehicle.
**Why it exists:** the output of the optimization engine (OR-Tools).

### 8. `vehicles` — delivery capacity *(simulated, Week 3)*
**Purpose:** the trucks/vans that execute routes.
**Important columns:**
- `vehicle_id` (PK).
- `capacity_kg`, `capacity_volume`, `home_warehouse_id` (FK), `status`.
**Relationships:** one vehicle → many routes.
**Why it exists:** routing must respect how much a vehicle can carry.
**Not in Olist — simulated.**

### 9. `disruptions` — traffic/weather events *(simulated, Week 7)*
**Purpose:** real-world conditions that delay or block routes.
**Important columns:**
- `disruption_id` (PK).
- `type` (traffic / weather / congestion), `affected_region`,
  `start_time`, `end_time`, `delay_factor`.
**Relationships:** affects routes (and indirectly orders).
**Why it exists:** lets us test how the system replans under stress.
**Not in Olist — simulated.**

### 10. `agent_decisions` — autonomous actions *(Week 5)*
**Purpose:** an audit log of what the CrewAI agents decided and why.
**Important columns:**
- `decision_id` (PK).
- `agent_name`, `decision_type` (reroute / restock / reassign),
  `related_order_id` (FK), `reasoning`, `timestamp`.
**Relationships:** references orders/routes/inventory it acted on.
**Why it exists:** transparency — we can see and evaluate autonomous behavior.

## Real vs. simulated (at the database level)

- **Real, from Olist:** `customers`, `products`, `warehouses` (location),
  `orders`, `order_items`.
- **Simulated/generated later:** `inventory`, `vehicles`, `disruptions`,
  and the computed `routes` / `agent_decisions`.

This mirrors the "real vs. simulated" split established in Week 0 — the real
Olist data is the trusted foundation, and the operational layer is added on top.

- **Week 2** populates `inventory` via simulation.
